        Frequently Asked Questions

        Please review these items to help troubleshoot common problems.

        Web Servers

        The dashboard fails to connect to the terminal, cannot be refreshed, and eventually crashes.

        These are the symptoms of a corrupted network buffer in the operating system kernel. Under these conditions some minor network calls, like an HTTP call to a web page or CSS file will succeed soon after web server launch but everything after fails to connect. The solution is to reboot the computer.

        On Windows I can create servers using ports 80 and 443, but it keeps resulting in an error on Linux.

        Linux, by policy, restricts normal users from running services on ports below 1025. This can be circumvented by running this application with elevated privilege, as an OS service, or using the package libcap. My preference is to run the application as an OS service using systemd.

        I created a redirect_asset rule for an absolute path, but it doesn't work.

        For security reasons server paths are absolute to the server's asset location on the machine file system and not the root of the file system. If redirection should occur for assets outside the server's asset tree I recommend creating a symbolic link in the server's asset location. In Linux this is accomplished with ln -s /desired/path link_name where the first path is where the files actually are and the second is where the link will be in the file system.

        I am trying to redirect traffic to a vanity domain from one server spawned by this application to another server spawned by this application, but it isn't working.

        The destination server must have the vanity domain listed in its domain_local list. For increased security servers will only accept connections to known addresses, which includes IP address and wildcards already supported on local machine interfaces.

        How can I ensure an end-to-end encrypted tunnel between my users and a proxied destination?

        First, verify the given server is set to encryption values both or secure. If the given server does not have a secure TLS instance running it will not be able to proxy to an encrypted TLS destination.

        Second, ensure there is a redirect_domain entry for the given server named with a .secure suffix, such as my-bank.com.secure as opposed to just my-bank.com. That tells this application to only proxy via TLS on incoming TLS connections.

        On a server entry I specified a redirect_domain entry using .secure as a naming suffix to create an end-to-end encrypted connection, but it does not work.

        First, verify the given server is set to encryption values both or secure. If the given server does not have a secure TLS instance running it will not be able to proxy to an encrypted TLS destination.

        Second, verify the destination is working without using a proxy connection by directly going to the destination address and port in a web browser. If the destination fails via direct request then a proxy connection will also expect to fail.

        Third, if the destination is failing through direct encrypted access via HTTPS but still works through unencrypted HTTP access you can fall back to partial encryption by deleting the redirect_domain entry with the .secure suffix. Partial encryption still encrypts data between the proxy, your server running from this application, and the users. Partial encryption will not encrypt data between the proxy and the destination. Typically partial encryption is good enough for privacy, but should not be trusted with sensitive data like passwords or connections to financial institutions.

        Containers

        None of the Docker commands seem to work, as though they aren't even there. I verified Docker and Docker Compose are installed.

        Most of the docker commands require elevated privilege to run and otherwise pretend to not exist. Docker does provide a guide to run docker commands as a regular user on Linux, but it seems many people (myself included) cannot get this to work.

        For security reasons I strongly recommend not executing docker commands directly with sudo unless you have no other choice. Instead I recommend running docker commands via some interface that is already running as a service. My personal solution is to run this application via systemd on Linux and then run Docker command via the terminal provided by the browser dashboard.

        I added a new Docker Container but it's showing offline. I also ran docker ps in the terminal and the new container is not in the list.

        Because the Docker Daemon is a service unrelated to this application containers require a separate activation step.

        After the new container is submitted click on the new container that now displays in the container list. Once expanded there will exist an Edit button under the Docker Compose text area. Click the Edit button and then click the Activate button that appears. This button will run the equivalent of Docker's up command and dump the output to the terminal.

        Docker wrote files as user root and now I cannot open them.

        This happens because by default Docker executes with elevated privilege. It is strongly recommended to specify in each Docker Compose file environmental properties PUID and PGID with values of 1000. This allows the container to execute as the OS's default user without elevated privilege access.

        On regular intervals my container appears to flicker between up and down. Upon further investigation I found the container issues 3 events: exec-create, exec-start, exec-die.

        This is how containers execute health checks. To stop a service from issuing a health check add the following property to the concerned service in the respective Docker Compose file: healthcheck: disable: true

        Docker isn't working after computer restart.

        See if the problem is solved by restarting service containerd first and then service docker second. Then review service logs to determine if service docker started without error.

        Terminal

        Why is my preferred choice of shell is not listed in the select list?

        Linux provides a catalogue of all installed shells in the file /etc/shells. Not all shells may add an entry to file upon their installation.

        Windows provides no such catalogue. Instead I must guess at the default locations for the most commonly used shells on Windows.

        To add additional options for the select list scroll to the bottom of the /lib/utilities/vars.ts file of this project looking for the vars.terminal property. Please ensure all entries are absolute file system paths to the local executable for any desired shell application.

        HTTP Test

        What are the common error states demonstrated by this HTTP Test tool?

        • Some domains are very slow to return a response and can take up to 10 minutes.
        • Some domains drop the connection or return an erroneous response. This could be due to a misconfiguration at the server, but is more typically an intentional way for servers to abandon traffic they don't want.
        • Some domains cause an EPROTO error. This is a TLS protocol negotiation error. It occurs because they server is using unsupported TLS 1.1 certificates, an error in this application, or mostly likely due to an error in Node.js. Examples of domains causing this error: prettydiff.com, www.army.mil, www.treasury.gov

        What is the fastest observed response from a public domain?

        The fastest I have seen consistently is www.microsoft.com which completes a roundtrip in about 0.065 seconds.

        WebSocket Test

        Why does this not connect if a port number is specified but otherwise connects to the same destination?

        The most likely answer is encryption mismatch, which is to say you are either attempting an encrypted connection to an insecure service or the opposite. This will absolutely occur if a port is specified for the destination, but may not occur if the port number is not provided. That is because the destination will likely direct your socket to the correct service location either by redirection or by implicit default port number.

        Why is it that some servers impose strange behaviors on WebSocket headers that don't work anywhere else?

        The conventions specified in RFC 6455 provide the common requirements for interconnection and messaging so that unrelated systems can talk in a commonly agreed upon way. When inter-connectivity between different unrelated systems is not necessary, such as one application talking to a different instance of the same application, then the application developer can modify the connection and messaging conventions as they wish.

        File System Information

        File System Objects

        block device
        A block device refers to an artifact representing a hardware device written to fixed-sized blocks, such as optical disks or flash memory.
        character device
        A character device is a simulated file that provides written access to hardware via streams, which allows direct access without buffering or fixed-size block segmentation. The typical use provides access to hardware via file system operations where performance is more important than storage integrity such as with serial ports, keyboards, cameras.
        directory
        A directory serves as a grouping mechanism allowing containing of child file system artifacts, which thereby provides a tree structure with organizational depth.
        FIFO device
        FIFO, or first in and first out, provides for pipe functionality. A pipe is an inter-process communication stream whereby one process sends its output directly to a different process as streamed input without help from an intermediary, such as a shell or user interaction. A FIFO allows for such inter-process communication via file system operations.
        file
        Files are the typical storage unit of file systems whereby text or binary is stored directly as opposed to a grouping or communication structure.
        socket
        File system sockets are similar to network sockets and FIFO devices. File system sockets exist at a low level in the kernel like network sockets and provide streamed inter-process communication like FIFO devices. File system sockets initiate using file system operations, but unlike FIFO devices are not necessarily subject to file system operations for transmitting data. Nonetheless file system sockets respect access and permission limitations inherited from their location on the file system.
        symbolic link
        A symbolic link is a virtual file that provides redirection within a file system. A symbolic link points to a file system artifact elsewhere on the file system and takes the identity of that artifact so as to function as a shortcut but yet appear exactly as the type of thing it points to. Since a symbolic link looks exactly like the thing it points to in both identity and function it cannot be discerned from a different file system type unless intentionally traversing and scanning for symbolic links.

        Permissions

        File system permission follow a 4 digit model. The digits follow this definition:

        1. The first digit determines whether to apply one of setuid, setgid, or sticky bit.
        2. The second digit provides a permission value for artifact owner.
        3. The third digit provides a permission value for assigned group of the artifact owner.
        4. The final digit provides a permission value for users that are not owners and not members of the owner's group(s).

        Permission values for the final 3 digits follow a combination of these values:

        • 4 = read permissions (r)
        • 2 = write permissions (w)
        • 1 = execute permissions (x)
        • 0 = no permissions

        So, for example, a value of 5 means 4 + 1, or read and execute permissions and a value of 7 means read, write, and execute permissions. A complete permission value of 0755 means owners have complete permissions but users and members of the owner's group(s) can only read and execute to that artifact.

          Hash Algorithms Explained

          Hash functions allow for generation of a predictable but challenging to reverse string token from a given string or binary input. Hashes are typically used as a form of identity or integrity check. A hash function provides a means of accepting input, whether string or binary, and returns a short randomly generated string token. The same input must always produce the same output, however.

          The strength of a hash function is the amount of work required to generate a collision. A collision occurs when two different forms of input generate the same output. MD5 and SHA1 hash algorithms, and all hash algorithms based upon them, are considered broken because there now exists several forms of known collisions and the work to generate new collisions against MD5 is trivial.

          One of the stronger modern hash algorithms is SHA3-512, which generates a fixed length 512bit token. The actual key space of that algorithm, the number of maximum potential values, is not known. The maximum possible key space based upon token size, 128 hexadecimal characters or 16^128, is larger than the number of atoms in the observed universe containing billions of galaxies each comprising billions of star systems each comprising billions of objects each comprising billions of atoms and so on.

          The servers feature fully capable, though optional, support for proxies.           HTTP Test

          A means to write arbitrary HTTP requests and read the raw response.

          HTTP Format (RFC 2616)

          HTTP is a text-based application layer transmission protocol defined as RFC 2616.

          The first line of an HTTP request a space separated list of method, resource path (also known as end point), and protocol. Each following line is separated by a CRLF (x0Dx0A) sequence. This application handles that formatting for you. The header portion of the request is separated from a body portion by an empty line, which is two or more consecutive sequences of CRLF. A body portion is optional and only used by certain HTTP methods. All lines after the first and before the body portion are headers. Each header gets its own line and is a case-insensitive name followed by a colon, zero or more spaces, and a text value.

          A host header is required and is generally the second line of the raw request string. The host value is taken from the domain portion of a web address. The domain portion of an address includes authentication, domain name, and port. The authentication portion is optional and rarely used.

          The port portion of a domain is optional and rarely included if connecting with plain text HTTP over port 80 or encrypted HTTPS over port 443. In all other cases the port number is a colon separated suffix to the domain name. In the case of an IPv6 address the domain name must be inclosed in square braces before specifying the port number, example: host: [2600:1700:1234:5678::a]:8081.

          • The method is one of the HTTP request methods. GET is the most common.
          • The end point looks like a Unix absolute file path always starting with a forward slash.
          • The protocol is HTTP, forward slash, and a version number, such as HTTP/1.1.

          An HTTP request can be either plain text or a TLS encryption. That distinction is not provided in the HTTP header but is inferred by context.

          The default behavior of HTTP version 1.1 is to keep a socket open for multiple sequential request/response round trips. HTTP 1.0 does not have this capability, called keep-alive, and it forbidden in HTTP/2 and HTTP/3. As a result, in HTTP/1.1, we must have an indicator different than socket termination to determine when a response completes. This following list determines response message completion.

          • The socket closes.
          • The response header is completed with the "\r\n\r\n empty line sequence and the response contains no body portion.
          • The response header contains a heading transfer-encoding: chunked and the body terminates with a message chunk containing exactly 0\r\n\r\n.
          • The response header contains a heading content-length with a numeric value and the byte length, not character length, of the body portion only equals that header value.

                Definitions of Server Object Properties

                activate (string)

                Instructs the application to bring the server online automatically at application start and server creation.

                block_list (string array)

                The block list contains three optional string arrays of criteria to determine if matching sockets should be instantly destroyed. At this time wildcards, CIDR notation, and ranges are not supported.

                domain_local (string array)

                Sockets featuring HTTP request host names in this list will be served from this server. Sockets featuring HTTP request host names not in this list or in the redirect_domain list will be destroyed.

                encryption (string)

                Whether to create an encrypted server for HTTPS and WSS protocols, an unencrypted server for HTTP and WS protocols, or both. Accepted values are both, open, or secure.

                http (object)

                File system paths to external utilities for handling the supported HTTP methods: delete, post, and put. HTTP request bodies are passed into an environmental variable on the respective child process named 'payload'.

                name (string)

                The name of this server.

                ports (object)

                The ports on which the server operates.

                redirect_asset (object)

                Redirects resource paths of HTTP requests within the given server based upon the hostname in the HTTP request header. This is useful for dynamic or vanity endpoints on a given server that wish to return something different than a relative file system destination. Each instance uses the structure of "domain": {"resource location": "new location"} because each server can represent 0 or more domain names.

                For example there could be an informational dashboard on some server at address: http://myexample.com/api_services/informational/dashboards/uptime/. For the sake of simplicity you want to serve that dashboard from an internal vanity domain like: http://api.dashboard/. You would need a rule to redirect from resource / to actual resource at /api_services/informational/dashboards/uptime, and that rule would look like: "api.dashboard": {"/": "/api_services/informational/dashboards/uptime/"}

                Asset redirection supports use of wildcards by specifying an asterisk at the end of asset locations. For example, redirection of request for /api_services/informational/dashboards/uptime/images/interface/title.png to /api/images/interface/title.png would occur with rule: "api.dashboard": {"/api/*": "/api_services/informational/dashboards/uptime/"} With wildcards all asset requests matching the address up to the asterisk will be redirected to the specified location.

                Please note the terminal slash in the prior examples. Without the terminating slash the redirected address becomes /api_services/informational/dashboards/uptimeimages/interface/title.png.

                redirect_domain (object)

                Redirects traffic to a different server location, such as a vanity domain that may or may not be served from the same server. Each instance uses the structure of "domain": ["new domain", port_number] because each server can represent 0 or more domain names. If the 'new domain' is an empty string the requested domain will be used. If the port number is not a number or 0 the current server's port number will be used.

                If TLS connections needs to be directed to a port different than open connections then use the naming convention myDomain.secure, for example: "jellyfin.x.secure": ["", 3003]

                single_socket (boolean)

                Providing property single_socket with value true will create a server that only accepts a first successful HTTP or WebSocket connection. All other connections will be ignored and the server will automatically remove itself once that connection completes. This server will never writing configuration data and so it will be forever lost once this application closes.

                temporary (boolean)

                Providing property temporary with value true will create a server that never writes configuration data. This server will be forever lost once this application closes.

                  Shells, Terminals, and Pseudo-Terminals

                  To provide human interaction directly to modern operating systems these modern operating systems include shells, terminals, and pseudo-terminals.

                  Shells, also called command shells, provide a means for human users to send instructions to operating systems directly without need to interact through other application processes. Human users, however, do not typically interact directly with command shells. The job of a shell is to provide a means to write commands, and optionally programmable automation, and that the operating system interprets and possibly responds to.

                  All the inputs and outputs through command shells occurs in raw text characters, as in text characters that can be typed directly from a physical keyboard. The reason humans do not typically engage with shells directly is because much of this raw text contains formatting and instructions and signals for interaction. These control sequences are comprised of raw text that greatly impedes the other raw text meant for humans to read. To properly interpret these formatting and control sequences, such as colors and focus movement, humans indirectly interact with shells through terminal applications.

                  A command terminal is an application that interprets shell output and provides a friendlier interface of human input to command shells. In other words shells are for computers to achieve human interaction and terminals are for humans to achieve shell interaction. Command terminals are sometimes referred to as tty devices, which refers to teletypewriter like an old typewriter for phone lines.

                  If shells are for computers and terminals are for humans then how do other applications interact with command shells? That is the job of pseudo-terminals, which are sometimes referred to as pty devices. Pseudo-terminals do the same job as terminal applications, but instead of graphically rendering text into pretty colors they send more friendly binary messages to consuming applications. For example when a user connects to a remote computer using SSH and sees a command shell they are interacting through pseudo-terminals on each side of the network with one side talking to a shell and the other side talking to a terminal for the human user.

                  Linux provides a functional API to engage with a PTY. This requires writing a C language file that makes use of functions open_pty or fork_pty from file pty.h and then compiling these custom instructions to binary. Windows provides its own functional API via the ConPTY architecture with functions in C# language. That requires writing a C# file that makes use of the Microsoft functions and then compiling to binary.

                  To solve for this in JavaScript world Microsoft maintains a package called node-pty that provides these binaries for both Windows, Linux, and FreeBSD. This application makes use of node-pty via package Xterm.js, as does every other application that wishes to provide a terminal interface to a web browser.

                  WebSocket Test

                  A tool to open and experiment with arbitrary WebSocket messaging.

                  WebSocket Protocol (RFC 6455)

                  WebSockets exist in two phases, a handshake phase to establish connection and then a messaging phase.

                  Handshake

                  The handshake phase resembles an HTTP GET request, wherein a RFC 2616 compliant request is sent from a client to a server. This request includes a vaguely random 32bit key compatible with BASE64 syntax. The request also allows for multiple points of identity, such as: a URI path, subprotocol names via comma separated values to the sec-websocket-protocol header, and custom headers.

                  Once the server receives the handshake request it responds with an algorithmic transform of the key provided by the request. The client then opens a persistent TCP socket to the server upon confirming the server's key transform.

                  Messaging

                  Messaging occurs in binary form. A single message may be divided into multiple transmissions called frames. Each frame is composed of a frame header of 2 to 14 bytes immediately followed by the frame message.

                  Message in WebSockets is also full-duplex. This means a single socket comprises two separate and fully independent communication channels allowing each side of the socket to send messages without waiting on the other side.

                  This application represents frame headers using an object of type name websocket_frame that comprises these properties:

                  extended
                  A number value storing the extended frame content length.
                  fin
                  A boolean value indicating whether the given frame is the final frame of a message.
                  len
                  A number value storing the abridged frame content length.
                  mask
                  A boolean value indicating whether the frame applies message masking.
                  maskKey
                  The 32 bit masking key used to unmask the frame body if the frame body were masked.
                  opcode
                  A number value determining frame function or message body format.
                  rsv1
                  A boolean value representing an optional bit flag that can be set for custom application preferences. This is not used by web browsers.
                  rsv2
                  Same as rsv1.
                  rsv3
                  Same as rsv1.
                  startByte
                  A number value only used internally by this application to indicate frame header length, and thus start position of the frame content body.

                  For frame content lengths of less than 126 bytes their lengths are literally represented in the len property of the frame header and then extended property receives a value of 0. For frame content lengths greater than 125 bytes but less than 65,536 bytes the len property receives a value of 126 and the extended property stores the actually frame content length occupying exactly 2 bytes of storage. For frame content lengths greater than 65,535 bytes the len property receives a value of 127 and the extended property stores the actual frame content length occupying exactly 8 bytes of storage.

                  The opcode property stores a 2 byte integer, values of 0 to 15. The first half of values indicate message format and the second half of values indicate control frames. The value definitions are as follows:

                  0
                  A continuation frame, which is any frame after the first representing a message divided amongst multiple frames.
                  1
                  A text frame, which stores messages in a text compatible string format of unspecified string encoding.
                  2
                  A binary frame, which means the content comprises a binary buffer.
                  3-7
                  Reserved for future use.
                  8
                  A close control frame. This lets the remote end know the socket will immediately self-destruct.
                  9
                  A ping control frame. This provides heartbeat to the remote end.
                  10
                  A pong control frame, which is just a response to a ping frame.
                  11-15
                  Reserved for future use.

                  Opcode values in the reserved for future use category can be safely repurposed for custom application use so long as both sides of the socket agree on these custom definitions.

                  In WebSockets messages cannot be intermingled. Each message must be sent serially in the order which the transmission buffer receives them. The WebSocket protocol does not specify a message queue, but I strongly recommend always including a message queue on each side of the socket. A message queue will ensure messages are not dropped if awaiting a prior message to finish transmission.
